VLANs - Layer 2 Security

OSI Layer 2: Data Link Layer

What are VLANs?

Virtual Local Area Networks (VLANs) are a powerful Layer 2 security mechanism that logically segments network traffic, even when devices share the same physical infrastructure.

How VLANs Enhance Security

  • Network Segmentation: Isolate different departments, user groups, or device types into separate broadcast domains
  • Broadcast Control: Limit broadcast traffic to specific VLANs, preventing network-wide exposure
  • Access Control: Restrict communication between VLANs, requiring routing through controlled points
  • Reduced Attack Surface: Compromised devices in one VLAN cannot directly access devices in another VLAN
  • Guest Network Isolation: Keep guest users separated from corporate resources

Example Configuration:

! Creating VLANs on a switch
vlan 10
  name CORPORATE
vlan 20
  name GUEST
vlan 30
  name SERVERS

! Assigning ports to VLANs
interface GigabitEthernet0/1
  switchport mode access
  switchport access vlan 10
Security Benefit: Even if an attacker gains access to the guest network (VLAN 20), they cannot directly communicate with corporate workstations (VLAN 10) or servers (VLAN 30) at Layer 2.

Firewalls - Layer 4 Security

OSI Layer 4: Transport Layer

What are Firewalls?

Firewalls operate at Layer 4 (and above) to control network traffic based on port numbers and protocols, allowing granular control over which services are accessible.

Layer 4 Service Control

At Layer 4, firewalls can distinguish between different services running on the same device by examining TCP/UDP port numbers:

  • SSH (Port 22): Secure remote access protocol
  • HTTP (Port 80): Web traffic protocol
  • HTTPS (Port 443): Encrypted web traffic
  • And many more... Each service uses specific ports

Allowing HTTP While Denying SSH

A common security scenario is to allow public access to web services (HTTP/HTTPS) while blocking remote administration (SSH) from untrusted networks.

Example: Linux iptables Configuration

# Allow HTTP traffic (port 80)
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Allow HTTPS traffic (port 443)
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Deny SSH from external networks (allow only from internal)
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j DROP

# Set default policy
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

Example: Windows Firewall (PowerShell)

# Allow HTTP inbound
New-NetFirewallRule -DisplayName "Allow HTTP" `
  -Direction Inbound -Protocol TCP -LocalPort 80 -Action Allow

# Allow HTTPS inbound
New-NetFirewallRule -DisplayName "Allow HTTPS" `
  -Direction Inbound -Protocol TCP -LocalPort 443 -Action Allow

# Block SSH from external, allow from internal network
New-NetFirewallRule -DisplayName "Allow SSH Internal Only" `
  -Direction Inbound -Protocol TCP -LocalPort 22 `
  -RemoteAddress 192.168.1.0/24 -Action Allow

New-NetFirewallRule -DisplayName "Block SSH External" `
  -Direction Inbound -Protocol TCP -LocalPort 22 -Action Block

Example: Cisco ASA Firewall

! Allow HTTP and HTTPS from outside to DMZ web server
access-list OUTSIDE_IN permit tcp any host 203.0.113.10 eq 80
access-list OUTSIDE_IN permit tcp any host 203.0.113.10 eq 443

! Deny SSH from outside (implicit deny)
! SSH access only allowed from inside network
access-list INSIDE_IN permit tcp 192.168.1.0 255.255.255.0 any eq 22

access-group OUTSIDE_IN in interface outside
access-group INSIDE_IN in interface inside
Security Benefit: Web services remain publicly accessible for legitimate users, while SSH access is restricted to trusted networks, significantly reducing the attack surface for remote exploitation.

Layered Security Approach

Defense in Depth

Combining VLANs (Layer 2) and Firewalls (Layer 4+) creates a robust, multi-layered security architecture:

Layer 2: VLAN Segmentation

Separates network segments to prevent lateral movement

?

Layer 3: Routing Control

Controls inter-VLAN communication through routers/firewalls

?

Layer 4: Service Filtering

Granular control over which services are accessible

Real-World Use Case:

  1. VLAN Segmentation: Separate corporate users, guest WiFi, and servers into different VLANs
  2. Inter-VLAN Routing: Force all VLAN-to-VLAN traffic through a firewall
  3. Firewall Rules: Allow guests to access web servers on HTTP/HTTPS but deny SSH access completely
  4. Result: Guests can browse websites, but cannot attempt SSH brute-force attacks on servers

Security Best Practices

VLAN Best Practices

  • Use separate VLANs for different security zones
  • Implement private VLANs for additional isolation
  • Disable DTP (Dynamic Trunking Protocol)
  • Use 802.1X for port-based authentication

Firewall Best Practices

  • Follow the principle of least privilege
  • Default deny, explicitly allow only necessary services
  • Regularly review and audit firewall rules
  • Use stateful inspection when possible
  • Log denied connections for security monitoring
